README Reflection Responses

1. How do I interpret user needs and implement them into a program? How does creating “user stories” help with this?

Interpreting user needs begins with understanding the problem from the user’s perspective rather than focusing immediately on a technical solution. Users often describe what they believe they want, but effective development requires identifying the underlying goal and value behind their request. I approach this by reviewing requirements carefully, asking clarifying questions, and identifying the intended outcome.

Creating user stories helps structure this process by framing requirements in a user-centered format: “As a [user], I want [goal], so that [reason].” This format ensures that each feature is tied directly to user value. User stories also break large requirements into smaller, manageable tasks that can be prioritized in a sprint backlog. Adding acceptance criteria further reduces ambiguity and provides measurable completion standards. Overall, user stories help translate abstract requirements into actionable development steps aligned with user needs.

2. How do I approach developing programs? What Agile processes do I hope to incorporate into my future development work?

My development approach has evolved from a linear model to an iterative and incremental mindset. Instead of attempting to design a complete system upfront, I now focus on delivering a functional version first and refining it over time. I begin by reviewing requirements, identifying user stories, prioritizing tasks, and organizing them into achievable sprint goals.

The Agile processes I plan to continue incorporating into my development work include:

• Sprint Planning to define clear and realistic objectives  
• Daily progress check-ins to track development and identify blockers  
• Sprint Reviews to demonstrate working functionality  
• Retrospectives to reflect on strengths and areas for improvement  
• Backlog refinement to maintain clarity and priority alignment  

Agile supports flexibility, accountability, and continuous improvement. As a working student, this structured yet adaptable framework helps me balance responsibilities while maintaining steady progress.

3. What does it mean to be a good team member in software development?

Being a strong team member involves more than technical skill. It requires effective communication, accountability, collaboration, and openness to feedback. A good team member keeps others informed of progress, communicates blockers early, and actively participates in discussions.

Through this course, I learned that transparency builds trust within a development team. Updating task statuses, participating in sprint ceremonies, and approaching code reviews with a growth mindset all contribute to team success. Software development is collaborative, and delivering value requires collective effort rather than individual recognition.

Overall Reflection

Completing the Sprint Review and Retrospective reinforced that Agile is both a structured framework and a mindset centered on adaptability and continuous improvement. This project strengthened my ability to translate user needs into working software, prioritize tasks effectively, and reflect critically on development practices. These skills will continue supporting my academic and professional growth in computer science.
